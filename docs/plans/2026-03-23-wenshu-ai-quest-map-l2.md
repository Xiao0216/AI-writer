# 文枢AI《项目全栈落地闯关大纲》第二层教案

> 本文是第一层 Quest Map 的教案扩展版。
>
> 使用方式：
> - 第一层 Quest Map 负责定义学生“这一关做什么”。
> - 第二层教案负责定义学生“这一关要交什么、会卡在哪里、考官怎么判分”。

## 阶段 1：脚手架与统一测试环境

### 🟢 Quest [1.1]：创建 Monorepo 根工作区
* **🎯 关卡目标**：初始化根目录工程，并准备统一脚本入口。
* **📦 学生提交物**：
  - 根目录 `package.json`
  - 可运行的根脚本列表截图或终端录屏
  - 项目目录结构截图
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生把前后端脚本都写到各自目录，根目录没有统一入口。
    - 助教提示：提醒学生“考官只看黑盒动作，根目录必须有一键入口”。
  - 卡点 2：学生一开始就创建大量业务文件。
    - 助教提示：要求删除超纲业务文件，只保留工程骨架。
* **🏁 评分 Rubric**：
  - **优秀**：根脚本命名清晰，目录结构整洁，学生能解释每个顶层目录的职责。
  - **合格**：根脚本可用，前后端目录存在，虽然命名略乱但不影响继续闯关。
  - **不通过**：只能分别进入子目录启动，根目录没有统一入口，或骨架混入大量业务代码。

### 🟢 Quest [1.2]：跑通前端空白站点
* **🎯 关卡目标**：让 `apps/web` 能独立启动，并显示一个最小可见页面。
* **📦 学生提交物**：
  - 前端启动终端截图
  - 浏览器打开首页截图
  - 学生自己的启动说明一句话
* **🧱 常见卡点与助教提示**：
  - 卡点 1：页面打开但白屏，没有任何文本锚点。
    - 助教提示：必须有“肉眼可见的识别文本”，否则考官无法验收。
  - 卡点 2：学生用了错误端口。
    - 助教提示：要求把端口固定到课程统一约定。
* **🏁 评分 Rubric**：
  - **优秀**：前端能稳定启动，页面可见，且学生能说清“为什么先做最小可见反馈”。
  - **合格**：页面可正常访问，有明确文本锚点。
  - **不通过**：服务起不来、端口不明确、页面没有可视反馈。

### 🟢 Quest [1.3]：跑通后端健康检查服务
* **🎯 关卡目标**：让 `apps/api` 能独立启动，并暴露一个最小可测的健康检查接口。
* **📦 学生提交物**：
  - API 启动日志截图
  - `curl /health` 返回结果截图
  - 健康检查 JSON 样例
* **🧱 常见卡点与助教提示**：
  - 卡点 1：后端启动了，但没有可测接口。
    - 助教提示：没有黑盒健康接口，就没有考官可见反馈。
  - 卡点 2：返回 HTML 而不是 JSON。
    - 助教提示：强调课程后端默认以 API 服务为中心。
* **🏁 评分 Rubric**：
  - **优秀**：接口返回稳定 JSON，字段命名清晰，学生理解健康检查的工程意义。
  - **合格**：`/health` 正常返回可见 JSON。
  - **不通过**：服务启动失败，或考官无法用命令验证。

### 🟢 Quest [1.4]：启动本地 PostgreSQL 与 Redis
* **🎯 关卡目标**：把本地基础设施真正跑起来，而不是停留在代码层。
* **📦 学生提交物**：
  - `docker compose ps` 输出截图
  - 容器运行状态截图
  - 基础设施说明一句话
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生本机没有 Docker 环境。
    - 助教提示：先解决运行条件，再进入数据层；不要用“以后再说”跳过。
  - 卡点 2：容器启动了但端口没暴露。
    - 助教提示：让学生先验证宿主机端口，而不是只看 Docker Desktop 的绿灯。
* **🏁 评分 Rubric**：
  - **优秀**：两个容器都能稳定运行，学生知道各自用途。
  - **合格**：容器已跑起来，端口可见。
  - **不通过**：只写了 compose 文件，没有真实启动证据。

### 🟢 Quest [1.5]：建立统一测试环境入口
* **🎯 关卡目标**：让学生能一键切换到 `test` 环境，而不是靠手敲临时环境变量。
* **📦 学生提交物**：
  - `apps/api/.env.test`
  - `apps/web/.env.test`
  - 测试环境脚本命令截图
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生直接把测试配置写死在代码里。
    - 助教提示：课程必须建立“配置与代码分离”的意识。
  - 卡点 2：前后端测试环境变量不一致。
    - 助教提示：要求学生明确“统一测试环境”的含义是前后端共同指向同一套资源。
* **🏁 评分 Rubric**：
  - **优秀**：前后端都有 `.env.test`，且脚本可以统一启动。
  - **合格**：测试环境文件存在，能被实际使用。
  - **不通过**：仍需手工拼环境变量，或前后端测试环境不统一。

## 阶段 2：静态信息架构与产品骨架

### 🟢 Quest [2.1]：搭出全站 App Shell
* **🎯 关卡目标**：渲染全站通用导航、页头和主内容区域。
* **📦 学生提交物**：
  - 三个页面的统一布局截图
  - 导航点击录屏
  - 页面框架结构说明
* **🧱 常见卡点与助教提示**：
  - 卡点 1：每个页面各写一套头部。
    - 助教提示：提醒学生“公共骨架”是工程训练，不是 UI 拼贴。
  - 卡点 2：导航能看不能点。
    - 助教提示：课程黑盒验收优先关注“能否操作”。
* **🏁 评分 Rubric**：
  - **优秀**：统一布局抽象得当，导航稳定。
  - **合格**：多页共享统一框架，路由能切。
  - **不通过**：页面各自为政，无法形成全站结构。

### 🟢 Quest [2.2]：渲染登录页和设置页静态骨架
* **🎯 关卡目标**：把“登录”和“个人设置”页面先做成可见原型。
* **📦 学生提交物**：
  - 登录页截图
  - 设置页截图
  - 表单字段列表
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生过早接 API，导致页面都还没成型就陷入联调。
    - 助教提示：本关目标是可见骨架，不是可用业务闭环。
  - 卡点 2：页面没有清晰的产品锚点。
    - 助教提示：要求学生页面上必须出现能说明该页用途的标题和按钮。
* **🏁 评分 Rubric**：
  - **优秀**：页面结构清楚，UI 已能支撑后续真实接线。
  - **合格**：核心区块齐全，可见可点。
  - **不通过**：只有零散组件，没有页面级结构。

### 🟢 Quest [2.3]：渲染“项目控制台”首页静态版
* **🎯 关卡目标**：把首页定义成“项目控制台”，而不是普通作品列表。
* **📦 学生提交物**：
  - 控制台首页截图
  - 页面模块标注图
  - 学生对“为什么不是作品列表”的一句解释
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生直接复用常规 dashboard 模板。
    - 助教提示：要求首页必须有“章节推进、热度、提醒、写回摘要”这类控制视角。
  - 卡点 2：页面信息太少，看不出产品方向。
    - 助教提示：让学生优先体现“控制站”的感觉，而不是追求精细视觉。
* **🏁 评分 Rubric**：
  - **优秀**：页面已经传达出“长篇控制台”的产品定位。
  - **合格**：至少具备明显区别于作品列表的控制模块。
  - **不通过**：本质上还是项目列表页换标题。

### 🟢 Quest [2.4]：渲染“项目真相区”静态版
* **🎯 关卡目标**：把标准控制文件做成一个可导航的静态界面。
* **📦 学生提交物**：
  - 真相区主界面截图
  - 各文档入口截图
  - 文件导航结构清单
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生把所有控制文件堆在一页里。
    - 助教提示：要求结构化导航，否则后续不可维护。
  - 卡点 2：文件名不统一。
    - 助教提示：强调课程里标准文件必须稳定，后面数据库和 AI 都要依赖它。
* **🏁 评分 Rubric**：
  - **优秀**：标准文件区已经有很强的信息架构感。
  - **合格**：主要控制文件都能被看到和切换。
  - **不通过**：控制文件缺失或结构混乱。

### 🟢 Quest [2.5]：渲染“章节控制页”静态版
* **🎯 关卡目标**：搭出正文页的三栏或双栏结构，让它看起来就像“写作工作台”。
* **📦 学生提交物**：
  - 章节控制页截图
  - 区域职责说明
  - 页面布局草图
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生只放一个编辑器，完全看不出控制能力。
    - 助教提示：至少要有正文区和控制区，不能退化成普通编辑器。
  - 卡点 2：侧栏太多，页面失去主次。
    - 助教提示：要求学生明确“正文区永远是主战场”。
* **🏁 评分 Rubric**：
  - **优秀**：写作台感非常清晰，产品方向可见。
  - **合格**：正文区和控制区都可见。
  - **不通过**：页面结构仍然像普通文档编辑器。

## 阶段 3：数据库与 Prisma 数据层

### 🟢 Quest [3.1]：初始化 Prisma 并连接测试数据库
* **🎯 关卡目标**：让后端真正能通过 Prisma 连接 `wenshu_ai_test`。
* **📦 学生提交物**：
  - `schema.prisma` 初始版本
  - `prisma generate` 成功截图
  - 测试数据库连接成功截图
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生只写 schema，不验证连接。
    - 助教提示：强调“能连上数据库”比“写了几行 schema”更重要。
  - 卡点 2：测试库和开发库混用。
    - 助教提示：让学生明确当前课程默认只操作测试库。
* **🏁 评分 Rubric**：
  - **优秀**：Prisma 与测试库连接清晰、配置整洁。
  - **合格**：可以生成 client 并连接数据库。
  - **不通过**：数据库未连接成功或环境混乱。

### 🟢 Quest [3.2]：落第一批核心表：用户、项目、章节
* **🎯 关卡目标**：先把最核心的三类实体建起来。
* **📦 学生提交物**：
  - migration 文件
  - 数据库表截图
  - 三张表的关系解释
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生把所有业务表一口气塞进去。
    - 助教提示：课程要求渐进式建模，不要一上来就全量 schema。
  - 卡点 2：关系字段不明确。
    - 助教提示：要求学生能口头解释“一个用户如何拥有多个项目和章节”。
* **🏁 评分 Rubric**：
  - **优秀**：核心实体边界清晰，关系设计稳定。
  - **合格**：三张表存在并能正确关联。
  - **不通过**：实体混乱，无法支撑后续业务。

### 🟢 Quest [3.3]：落第二批控制表：标准文件与动态状态
* **🎯 关卡目标**：把“项目真相文件”正式变成数据库里的核心对象。
* **📦 学生提交物**：
  - 控制文档相关 schema
  - 动态状态相关 schema
  - 数据库面板截图
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生还想把这些信息塞回项目主表里。
    - 助教提示：强调“真相文件是第一类对象，不是备注字段”。
  - 卡点 2：动态状态和日志不分。
    - 助教提示：区分“故事真相”和“审计记录”。
* **🏁 评分 Rubric**：
  - **优秀**：标准文件、动态状态、日志边界非常清楚。
  - **合格**：控制文档与项目主记录已拆分。
  - **不通过**：仍未建立控制文件数据层。

### 🟢 Quest [3.4]：落第三批扩展表：额度、支付、AI 生成、马拉松任务
* **🎯 关卡目标**：为会员、支付、生成日志、异步任务预留真实表结构。
* **📦 学生提交物**：
  - 扩展 migration
  - 表结构截图
  - 各表用途说明
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生因为暂时没做支付，就不建表。
    - 助教提示：这关的重点是产品数据边界，不是付款按钮。
  - 卡点 2：把 AI 生成结果只放日志，不考虑可追踪性。
    - 助教提示：要求学生为后续额度和审计埋好结构。
* **🏁 评分 Rubric**：
  - **优秀**：商业能力与基础写作能力的数据层边界清楚。
  - **合格**：支付、生成、任务三类表存在。
  - **不通过**：扩展表缺失，后续商业链路无法落地。

## 阶段 4：登录与身份体系

### 🟢 Quest [4.1]：实现开发环境快捷登录 API
* **🎯 关卡目标**：先让学生做出最小可用登录，便于后续所有前后端联调。
* **📦 学生提交物**：
  - 登录接口黑盒测试截图
  - 数据库中新增用户截图
  - 登录成功响应示例
* **🧱 常见卡点与助教提示**：
  - 卡点 1：接口能返回用户，但不写 cookie。
    - 助教提示：提醒学生“登录成功”不等于“能维持会话”。
  - 卡点 2：用户没有真正写入数据库。
    - 助教提示：要求考官必须能在数据库面板看到新增记录。
* **🏁 评分 Rubric**：
  - **优秀**：接口、cookie、数据库写入三者都闭环。
  - **合格**：接口成功，且用户记录落库。
  - **不通过**：只是返回假数据，无法支撑后续登录态。

### 🟢 Quest [4.2]：把前端登录页接到开发登录 API
* **🎯 关卡目标**：让学生第一次看到“点按钮真的能进系统”。
* **📦 学生提交物**：
  - 登录页操作录屏
  - 跳转到 dashboard 截图
  - 刷新后仍保持登录的录屏或截图
* **🧱 常见卡点与助教提示**：
  - 卡点 1：登录后跳转成功，但刷新丢失状态。
    - 助教提示：检查 `credentials: include` 和 cookie 传递。
  - 卡点 2：接口成功但前端没有错误提示逻辑。
    - 助教提示：要求失败时也有可见反馈。
* **🏁 评分 Rubric**：
  - **优秀**：登录体验完整，错误和成功路径都可见。
  - **合格**：可以登录并保持会话。
  - **不通过**：只能“看起来登录”，实际无法持续使用。

### 🟢 Quest [4.3]：实现“当前用户 + 个人设置”闭环
* **🎯 关卡目标**：打通 `auth/me` 和 `users/me`，让学生理解“登录后身份读取与资料编辑”。
* **📦 学生提交物**：
  - 设置页截图
  - 修改资料前后对比截图
  - 数据库中用户资料变化截图
* **🧱 常见卡点与助教提示**：
  - 卡点 1：表单能改本地状态，但没真正保存。
    - 助教提示：刷新后是否保留，是最简单的验收方式。
  - 卡点 2：读取的是假用户，而不是当前登录用户。
    - 助教提示：要求学生明确“当前用户”的来源。
* **🏁 评分 Rubric**：
  - **优秀**：读取、编辑、保存、刷新四步都稳定。
  - **合格**：能成功修改并持久化个人资料。
  - **不通过**：只改了前端显示，没有真正数据闭环。

### 🟢 Quest [4.4]：接入真实微信登录
* **🎯 关卡目标**：让学生把真实第三方登录链路接进现有系统。
* **📦 学生提交物**：
  - 微信授权跳转截图
  - 回调成功后的系统页面截图
  - 真实或测试 openid 用户记录截图
* **🧱 常见卡点与助教提示**：
  - 卡点 1：授权地址能跳，但回调后回不到前端。
    - 助教提示：先检查回调 URL，再检查前后端地址是否一致。
  - 卡点 2：学生把真实微信登录和开发登录混在一起写死。
    - 助教提示：要求分清“本地联调入口”和“真实登录入口”。
* **🏁 评分 Rubric**：
  - **优秀**：真实第三方链路完整、体验顺滑。
  - **合格**：能完成从授权到登录态建立的闭环。
  - **不通过**：只能跳授权页，无法完成回调闭环。

## 阶段 5：小说项目与标准控制文件

### 🟢 Quest [5.1]：打通新建项目表单与 API
* **🎯 关卡目标**：让用户可以创建一部小说项目，而不是只看到静态占位。
* **📦 学生提交物**：
  - 新建项目录屏
  - 数据库中新项目记录截图
  - 创建后跳转页面截图
* **🧱 常见卡点与助教提示**：
  - 卡点 1：学生用假数据渲染详情页，但数据库没写入。
    - 助教提示：考官一定要能在数据库里看到新项目。
  - 卡点 2：项目创建后不跳转，用户不知道成功没。
    - 助教提示：强调“操作后的物理反馈”。
* **🏁 评分 Rubric**：
  - **优秀**：创建流程顺滑，成功反馈清楚，数据真实落库。
  - **合格**：可创建项目并进入项目页。

### 🟢 Quest [5.2]：项目创建时自动生成 10 份标准控制文件

## 阶段 8：正文写作工作台与 AI 生成功能

## 助教通用带教原则

### 1. 不替学生跨关做事

助教最常见的错误，是在学生卡住时直接把下一关的内容提前塞给他。

正确方式：

- 只提示当前关卡的最小修复路径
- 不提前引入下一关的新概念
- 不用“以后还要做 X”压垮当前关卡

### 2. 永远要求可见反馈

如果学生说“我已经做完了”，助教必须追问：

- 浏览器上我能看到什么？
- 终端里我能看到什么？
- 数据库里我能看到什么？

没有物理反馈，就不算通过。

### 3. 优先保住正反馈

当学生连续两关都卡住时，助教应该：

- 先帮助他恢复最小可运行状态
- 再继续业务推进

不要让课程进入“工程债越积越多”的失控区。

## PM 考官总规则

你不需要看代码，只看这四类证据：

1. 浏览器可见页面反馈
2. 终端命令可见输出
3. 数据库面板可见记录
4. 测试命令可见通过结果

凡是不能通过这四类证据验证的关卡，都应该视为未完成。
