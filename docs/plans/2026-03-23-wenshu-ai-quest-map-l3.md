# 文枢AI《项目全栈落地闯关大纲》第三层课程包

> 本文是第一层 Quest Map 与第二层教案的继续下钻版本。
>
> 第三层课程包面向三类人：
> - 讲师：知道每关该怎么讲
> - 助教：知道学生会卡在哪
> - 产品经理 / 考官：知道每关的设计和工程是否真的对齐

---

## 第三层课程包使用规则

每个 Quest 在第三层统一增加四类字段：

1. `🎨 设计输入`：这一关开始前，学生应该已经在 Figma 中确认什么
2. `🧩 设计细节要求`：本关必须体现的交互、布局、视觉层级
3. `🎬 讲师演示脚本`：讲师上课时建议如何演示
4. `🧯 错题与救火提示`：学生最常犯的错误以及如何现场救火

在这套课程里，**先设计，后编码** 是硬规则。

如果某个开发关卡没有对应的 Figma 设计输入，就不允许学生进入实现阶段。

## 阶段 0：Figma 设计定稿

*这一阶段放在所有工程开发之前。目标不是做炫酷视觉，而是让学生先学会把 PRD 翻译成“可交付的界面与交互规格”。*

### 🟢 Quest [0.1]：建立 Figma 项目文件与页面结构
* **🎯 关卡目标**：创建课程统一的 Figma 文件，并按产品结构拆好页面。
* **🛠️ 推荐工具/轮子**：`Figma`, 页面命名规范, Section
* **🚫 边界限制 (防超纲)**：本关只做文件结构，严禁开始画高保真界面。
* **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：打开学生的 Figma 文件
  - [ ] 预期结果 1：能看到按阶段或页面组织的页面树，而不是所有画板堆在一页
  - [ ] 测试动作 2：检查页面命名
  - [ ] 预期结果 2：至少存在 `登录 / 控制台 / 项目真相区 / 章节控制页 / 会员页 / 马拉松控制页` 这类页面分区
* **📦 学生提交物**：
  - Figma 文件链接
  - 页面树截图
  - 页面命名说明
* **🎨 设计输入**：PRD 与产品手册 V2
* **🧩 设计细节要求**：
  - 文件结构必须体现产品信息架构
  - 页面命名必须可让 PM 一眼看懂用途
  - 不允许使用“Page 1 / Untitled frame”之类默认命名
* **🎬 讲师演示脚本**：
  - 先展示一个混乱 Figma 文件的反例
  - 再展示如何从产品结构推导设计文件结构
  - 最后强调“设计资产可维护性”与“工程资产可维护性”是同一件事
* **🧯 错题与救火提示**：
  - 错题 1：学生把所有页面放在同一张长画布上
    - 救火：要求按产品模块重新拆 Page 或 Section
  - 错题 2：只画最想画的首页，其他页面完全缺失
    - 救火：提醒学生课程是“系统交付”，不是单页美术展示

### 🟢 Quest [0.2]：完成低保真信息架构草图
* **🎯 关卡目标**：先用低保真草图确认页面结构与任务流，不陷入视觉细节。
* **🛠️ 推荐工具/轮子**：Figma Wireframe kit, Auto Layout
* **🚫 边界限制 (防超纲)**：本关只画线框，严禁上颜色系统与精细组件。
* **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：查看首页、项目页、章节页线框
  - [ ] 预期结果 1：能一眼看出区域分工，而不是一堆无标签方块
  - [ ] 测试动作 2：要求学生口头解释用户路径
  - [ ] 预期结果 2：学生能讲清“用户先看什么、再做什么、操作后去哪”
* **📦 学生提交物**：
  - 低保真线框图
  - 用户路径说明
  - 页面区域职责说明
* **🎨 设计输入**：产品手册中的核心页面结构
* **🧩 设计细节要求**：
  - 首页必须体现“控制台”而不是“列表页”
  - 章节页必须体现“正文区 + 控制区”
  - 会员页必须体现“能力分层”而不是单按钮支付页
* **🎬 讲师演示脚本**：
  - 先用白板讲用户主链路
  - 再把主链路翻成 3 到 5 个关键页面线框
  - 强调“线框的价值是做结构决策，不是做视觉创作”
* **🧯 错题与救火提示**：
  - 错题 1：学生在低保真阶段就开始抠配色和圆角
    - 救火：要求回到“区域职责”问题
  - 错题 2：学生的页面结构无法支撑产品逻辑
    - 救火：要求先修信息架构，再谈视觉

### 🟢 Quest [0.3]：定义设计系统基础 Token
* **🎯 关卡目标**：建立课程内统一的颜色、字体、间距、圆角、阴影规则。
* **🛠️ 推荐工具/轮子**：Figma Variables, Styles, Type Scale
* **🚫 边界限制 (防超纲)**：本关只做基础 token，严禁开始全量组件库建设。
* **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：查看 Figma 的颜色与文字样式
  - [ ] 预期结果 1：存在统一命名，而不是手工散落样式
  - [ ] 测试动作 2：抽查两个页面
  - [ ] 预期结果 2：视觉风格一致，不像两个产品
* **📦 学生提交物**：
  - 基础 token 页面截图
  - 颜色 / 字体 / 间距命名清单
  - 设计系统简表
* **🎨 设计输入**：低保真线框与产品定位
* **🧩 设计细节要求**：
  - 设计风格必须服务“长篇小说控制站”，不能像通用 SaaS 后台
  - 信息层级要明显，适合高密度信息场景
  - 字体和色彩需兼顾长时间阅读与控制台可读性
* **🎬 讲师演示脚本**：
  - 先解释什么叫 token
  - 再展示一个“没有 token 的页面为什么难维护”
  - 最后展示 token 如何反哺前端实现
* **🧯 错题与救火提示**：
  - 错题 1：学生在不同页面各自挑颜色
    - 救火：统一抽出颜色变量
  - 错题 2：排版层级混乱
    - 救火：回到标题、正文、说明文字三级结构

### 🟢 Quest [0.4]：产出高保真关键页面设计稿
* **🎯 关卡目标**：把关键页面做成可直接指导开发的高保真设计稿。
* **🛠️ 推荐工具/轮子**：Figma Components, Auto Layout, Interactive Components
* **🚫 边界限制 (防超纲)**：本关只覆盖关键页面，严禁为了完整性把所有边缘页都画完。
* **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：查看关键页面设计稿
  - [ ] 预期结果 1：至少包含 `登录 / 项目控制台 / 项目真相区 / 章节控制页 / 会员页`
  - [ ] 测试动作 2：放大查看页面细节
  - [ ] 预期结果 2：能看到真实布局、文字层级、按钮状态，而不只是草图
* **📦 学生提交物**：
  - 高保真关键页面链接
  - 关键页面导出截图
  - 页面说明文档
* **🎨 设计输入**：低保真线框 + 设计 token
* **🧩 设计细节要求**：
  - 登录页：区分微信登录与开发登录
  - 控制台：突出章节推进、热度、提醒、写回摘要
  - 项目真相区：有稳定导航结构
  - 章节控制页：正文区必须是视觉主区域
  - 会员页：突出能力分层与升级理由
* **🎬 讲师演示脚本**：
  - 演示如何从线框升级到高保真
  - 强调“每个组件都必须服务信息层级”
* **🧯 错题与救火提示**：
  - 错题 1：设计稿很漂亮，但不适合真实开发
    - 救火：要求学生补充状态、尺寸、空态、错误态
  - 错题 2：页面缺少主次节奏
    - 救火：回到用户第一眼该看什么的问题

### 🟢 Quest [0.5]：定义交互规格与状态清单
* **🎯 关卡目标**：在开发前把关键交互状态说清楚。
* **🛠️ 推荐工具/轮子**：Prototype arrows, annotations, flow notes
* **🚫 边界限制 (防超纲)**：本关只定义核心交互，严禁写完整前端实现方案。
* **✅ 考官验收标准 (PM Checklist)**：
  - [ ] 测试动作 1：查看登录、创建项目、生成控制卡、完成章节的交互说明
  - [ ] 预期结果 1：能明确看到成功态、失败态、加载态说明
  - [ ] 测试动作 2：让学生口头走一遍主链路
  - [ ] 预期结果 2：学生能从设计稿讲到页面跳转和状态变化
* **📦 学生提交物**：
  - 关键交互注释图
  - 状态清单文档
  - 主链路原型链接
* **🎨 设计输入**：高保真关键页面
* **🧩 设计细节要求**：
  - 每个关键动作必须有 loading / success / error 三态说明
  - 所有弹窗、保存、生成行为都要有物理反馈
  - 不允许把交互逻辑留给开发时临时猜
* **🎬 讲师演示脚本**：
  - 用“登录失败但页面没反应”的反例说明为什么交互规格重要
  - 再示范如何把一个按钮补齐三种状态
* **🧯 错题与救火提示**：
  - 错题 1：只画正常态，不画异常态
    - 救火：要求补失败态与空态
  - 错题 2：交互说明是大段自然语言，无法指导开发
    - 救火：改成可逐项核对的状态清单

## 阶段 1：脚手架与统一测试环境

### 🟢 Quest [1.1]：创建 Monorepo 根工作区
* **🎯 关卡目标**：初始化根目录工程，并准备统一脚本入口。
* **🎨 设计输入**：
  - 无需高保真设计稿
  - 但需要已有产品页面树，确保工程结构不偏离产品结构
* **🧩 设计细节要求**：
  - 根目录命名与项目分层要能映射到未来 Figma 页面与产品模块
* **🎬 讲师演示脚本**：
  - 演示“为什么同一个项目要先搭可维护骨架”
  - 对照 Figma 页面树讲工程目录分层
* **🧯 错题与救火提示**：
  - 错题：学生把目录按技术堆，而不是按应用边界拆
    - 救火：要求回到 `apps/web` 与 `apps/api` 两个交付面

### 🟢 Quest [1.2]：跑通前端空白站点
* **🎯 关卡目标**：让 `apps/web` 能独立启动，并显示一个最小可见页面。
* **🎨 设计输入**：
  - 阶段 0 的 Figma 页面树
  - 首页线框或低保真版本
* **🧩 设计细节要求**：
  - 最小页面也要体现产品名和基础品牌锚点
  - 页面不允许是完全无信息的默认模板
* **🎬 讲师演示脚本**：
  - 先启动空白 Next.js
  - 再把默认页替换成“最小产品壳”
* **🧯 错题与救火提示**：
  - 错题：学生直接保留 Hello World
    - 救火：要求替换成文枢AI品牌锚点

### 🟢 Quest [1.3]：跑通后端健康检查服务
* **🎯 关卡目标**：让 `apps/api` 能独立启动，并暴露一个最小可测的健康检查接口。
* **🎨 设计输入**：
  - 无界面设计要求
* **🧩 设计细节要求**：
  - 本关主要是工程可见性，不要求 Figma 对应稿
* **🎬 讲师演示脚本**：
  - 用终端和 curl 展示“黑盒可验收”的后端开发方式
* **🧯 错题与救火提示**：
  - 错题：学生想等前端做好后再管后端
    - 救火：强调后端同样要先有最小可见反馈

### 🟢 Quest [1.4]：启动本地 PostgreSQL 与 Redis
* **🎯 关卡目标**：把本地基础设施真正跑起来，而不是停留在代码层。
* **🎨 设计输入**：
  - 无
* **🧩 设计细节要求**：
  - 无界面要求，但必须服务后续设计落地
* **🎬 讲师演示脚本**：
  - 讲解为什么“页面设计得再好，没有基础设施也只是海报”
* **🧯 错题与救火提示**：
  - 错题：学生跳过基础设施，继续前端页面
    - 救火：暂停 UI 关卡，先恢复运行条件

### 🟢 Quest [1.5]：建立统一测试环境入口
* **🎯 关卡目标**：让学生能一键切换到 `test` 环境，而不是靠手敲临时环境变量。
* **🎨 设计输入**：
  - 无
* **🧩 设计细节要求**：
  - 虽无视觉稿，但要求测试环境命名与考官路径一致
* **🎬 讲师演示脚本**：
  - 展示同一套页面如何在 test 环境中稳定复现
* **🧯 错题与救火提示**：
  - 错题：学生使用多个不一致端口和环境名称
    - 救火：统一课程规范

## 阶段 2：静态信息架构与产品骨架

### 🟢 Quest [2.1]：搭出全站 App Shell
* **🎯 关卡目标**：渲染全站通用导航、页头和主内容区域。
* **🎨 设计输入**：
  - Figma 高保真导航规范
  - 关键页面的统一头部与主内容区定义
* **🧩 设计细节要求**：
  - 导航层级与 Figma 保持一致
  - 页头需有品牌锚点和主导航入口
  - 主内容区需留出后续高密度控制台空间
* **🎬 讲师演示脚本**：
  - 从 Figma 导航稿出发，先搭最小 App Shell
  - 再带学生切换 3 个页面，强调“共享骨架”
* **🧯 错题与救火提示**：
  - 错题：学生逐页复制粘贴导航
    - 救火：强制抽成共享布局组件

### 🟢 Quest [2.2]：渲染登录页和设置页静态骨架
* **🎯 关卡目标**：把“登录”和“个人设置”页面先做成可见原型。
* **🎨 设计输入**：
  - 登录页高保真稿
  - 设置页高保真稿
  - 关键按钮状态稿
* **🧩 设计细节要求**：
  - 登录页必须清晰区分微信登录入口与开发登录入口
  - 设置页需体现表单分区和保存操作区
  - 页面标题、说明文案、输入区域要有明显层级
* **🎬 讲师演示脚本**：
  - 先对照 Figma 标注页面区块
  - 再逐块落成静态结构
* **🧯 错题与救火提示**：
  - 错题：学生先写逻辑，导致页面结构失真
    - 救火：先把结构对齐设计稿，再接逻辑

### 🟢 Quest [2.3]：渲染“项目控制台”首页静态版
* **🎯 关卡目标**：把首页定义成“项目控制台”，而不是普通作品列表。
* **🎨 设计输入**：
  - 控制台高保真稿
  - 数据密度与层级稿
* **🧩 设计细节要求**：
  - 页面首屏必须突出“当前推进、提醒、热度、写回摘要”
  - 作品列表只能作为控制台中的一个区域，不能霸占首页
  - 信息卡片需要有主次与扫描顺序
* **🎬 讲师演示脚本**：
  - 先展示“错误的作品列表首页”
  - 再展示“控制台型首页”
  - 让学生理解结构比花哨更重要
* **🧯 错题与救火提示**：
  - 错题：学生把控制台做成四个无意义 KPI 方块
    - 救火：要求所有区块都回答“作者现在该干什么”

### 🟢 Quest [2.4]：渲染“项目真相区”静态版
* **🎯 关卡目标**：把标准控制文件做成一个可导航的静态界面。
* **🎨 设计输入**：
  - 真相区信息架构稿
  - 左侧导航或 tab 结构稿
* **🧩 设计细节要求**：
  - 标准文件入口必须稳定、可预期
  - 当前选中文档要有明显状态
  - 文档区需预留后续长文本编辑空间
* **🎬 讲师演示脚本**：
  - 先讲“真相文件为什么是产品核心”
  - 再带学生按设计稿搭结构
* **🧯 错题与救火提示**：
  - 错题：学生用弹窗承载文档编辑
    - 救火：要求用稳定工作区而非临时弹层

### 🟢 Quest [2.5]：渲染“章节控制页”静态版
* **🎯 关卡目标**：搭出正文页的三栏或双栏结构，让它看起来就像“写作工作台”。
* **🎨 设计输入**：
  - 章节控制页高保真稿
  - 编辑器区与控制区比例稿
* **🧩 设计细节要求**：
  - 正文区必须是视觉主区域
  - 控制卡、角色切片、伏笔切片的位置要符合设计稿
  - 操作按钮与状态条要贴近创作流而不是漂浮在角落
* **🎬 讲师演示脚本**：
  - 对照设计稿拆“主编辑区 / 控制侧栏 / 状态反馈区”
* **🧯 错题与救火提示**：
  - 错题：学生把所有控制模块都塞到正文上方
    - 救火：回到设计稿的视觉主次关系

## 阶段 3 到 阶段 11 的第三层补充规则

> 为避免文档无限膨胀，后续阶段采用统一补充规则：
> - 每一关必须先对照对应 Figma 设计稿或交互规格
> - 若该关为纯后端关卡，则必须对照前一阶段确认好的交互状态清单
> - 所有与页面相关的 Quest，都必须额外检查以下设计维度

### 所有页面类 Quest 的统一设计细节要求

1. 必须对齐 Figma 的：
   - 布局结构
   - 信息层级
   - 交互状态
   - 空态 / 错误态 / 加载态

2. 不允许开发时临时拍脑袋决定：
   - 按钮文案
   - 弹窗结构
   - 表单字段顺序
   - 页面主次层级

3. 每个与 UI 有关的 Quest，讲师都要额外演示：
   - 如何从 Figma 找到开发所需信息
   - 如何把一个组件拆成骨架、状态、数据接线三步

4. 助教在 UI 类关卡中必须追问：
   - 这一版是否和设计稿一致
   - 如果不一致，是有意优化还是开发偷懒
   - 学生是否能指出设计稿中对应位置

### 所有后端类 Quest 的统一设计补充要求

虽然没有直接界面，但每个后端关卡都必须回答一个问题：

**这个接口或数据结构，服务的是哪一个设计稿里的交互状态？**

如果学生答不上来，说明他是在“为技术而技术”，不是在实现产品。

## 第三层新增：讲师总演示路径

整套课程建议讲师按以下节奏演示：

1. 先讲产品手册
2. 再讲 Figma 设计稿
3. 再讲第一层 Quest Map
4. 再讲第二层教案要求
5. 最后进入编码演示

讲师永远不要直接从代码开始。

正确顺序必须是：

**PRD -> 信息架构 -> Figma -> 黑盒验收目标 -> 工程实现**

## 第三层新增：考官重点检查项

相较前两层，第三层特别要求考官检查：

1. 学生有没有在开发前完成 Figma 设计定稿
2. 页面实现是否真的对齐设计稿
3. 每个接口是否能对应到明确的交互状态
4. 学生是否理解“先设计、后实现”的产品落地顺序

如果学生绕过 Figma 直接开始写代码，即使页面能跑，也不应评为优秀。
