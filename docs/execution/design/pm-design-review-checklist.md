# PM 设计评审检查表

## 评审目的

确认学生是否已经完成“可以指导开发”的设计交付，而不是只交了一套好看的图。

## 检查项

### 文件结构

- [ ] Figma 文件按模块分区，而不是一页乱堆
- [ ] 命名稳定，没有 `最终版`、`新版2` 之类命名

### 信息架构

- [ ] 首页是控制台，不是列表页
- [ ] 项目真相区有稳定导航
- [ ] 章节页是工作台，不是普通编辑器
- [ ] 会员页有能力分层，而不是单支付页

### 关键页面覆盖

- [ ] 登录页
- [ ] 项目控制台
- [ ] 项目真相区
- [ ] 章节控制页
- [ ] 会员页

### 状态设计

- [ ] 登录有成功 / 失败 / 加载态
- [ ] 创建项目有成功 / 失败 / 加载态
- [ ] 生成控制卡有成功 / 失败 / 加载态
- [ ] 完成章节有成功 / 失败 / 加载态

### 可开发性

- [ ] 页面主次清楚
- [ ] 关键按钮位置明确
- [ ] 表单字段顺序明确
- [ ] 没有把交互逻辑留给开发时临时猜

## 结论规则

### 通过

全部关键页面齐全，状态明确，可直接进入 Quest 1。

### 有条件通过

页面齐全，但状态缺失或命名混乱，需要在进入开发前补齐。

### 不通过

如果以下任一情况出现，不能进入开发：

- 关键页面不全
- 章节页不是工作台结构
- 首页不是控制台结构
- 交互状态没有定义
- 文件结构混乱到无法对照开发
